⑤ 改进执行方案(可落地)

做什么 → 怎么做 → 省多少

🚨 优先级 0(风险级,先于一切提效):生产上线安全硬门禁 —— 杜绝「没验证就 commit 直发生产」

数据里最危险的不是耗时,而是 AI 改动没真正验证就进了发布流程:大量 submit_partial、preview_partial 冒烟有遗留、pod 未就绪却继续、/ai-deploy-test force 硬推。机器人类尤其致命(CrashLoop / import 崩溃 / Has 恒 true 检查永不执行 / 交 stub 占位符,而输出对错无自动判定)—— 一旦发布,线上机器人就乱回/漏检。

要加的硬门禁(缺一不可):
  1. 禁止 partial/best-effort 进发布:preview_partial/submit_partial/pod 未就绪/冒烟有遗留 一律判 FAIL,不得进入 deploy;/ai-deploy-test force 对生产禁用。
  2. 机器人服务:发布前必须过「行为评测门禁」:每个 robot 仓建离线评测集(输入→期望行为)+自动打分,评测不通过禁止合入/发布 —— 替代「预览起来了就算过」。
  3. 「测不了就不许改」:CI 检测到无 pytest/vitest 可运行环境(如 PEP668/无网/Go 版本不符)时,禁止 Agent 改启动路径/核心逻辑代码(用户在 #621 已提)—— 不让"没跑过测试"的改动蒙混过关。
  4. 运行时验证纳入硬门禁:预览必须真实拉起 + 冒烟真通过(看 body/接口真返回,不看 pod 状态)才算过;发布前 required review + 真集成测试绿。
  5. 发布=合入边界:未合并的实现 PR 不得发布(现已有此拦截,保持);禁止绕过 PR 直接 commit 到生产分支。

这条不省小时,但避免生产事故 —— 优先级高于下面所有提效项。

做得好的凭什么(可复制经验)

  1. issue 正文自带规格/诊断 →「照抄落地」(最强因子):给根因、修复 SQL、文件:行号、参考 PR、参数表,AI 1 轮就对(#1128/#1226/#1013/#837)。
  2. 单点小改 + 量化验收:单文件/单接口、验收一句话或明确数字(500ms/两位小数/65%),改动面小难翻车(#1358/#1241/#1056)。
  3. 有现成同构模板可抄:参考已有 PR / playground 已存在 / 照清单逐项(#608 参考PR#117、#680、#1013)。
  4. 仓库 CLAUDE.md/skill 覆盖到位:AI 一次照约定做对(#1068 读了 datastat CLAUDE.md+skill、#1048 配置驱动原则)。

核心洞察:做得好=「人把 spec 写清 或 AI 已有领域知识」。所以两条腿:①要求 issue 写清可落地 spec ②给 AI 领域知识(I1)让它不必每次靠人喂。
