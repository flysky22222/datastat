⑤ 改进执行方案(可落地)

做什么 → 怎么做 → 省多少

🚨 优先级 0(风险级,先于一切提效):生产上线安全硬门禁 —— 杜绝「没验证就 commit 直发生产」

数据里最危险的不是耗时,而是 AI 改动没真正验证就进了发布流程:大量 submit_partial、preview_partial 冒烟有遗留、pod 未就绪却继续、/ai-deploy-test force 硬推。机器人类尤其致命(CrashLoop / import 崩溃 / Has 恒 true 检查永不执行 / 交 stub 占位符,而输出对错无自动判定)—— 一旦发布,线上机器人就乱回/漏检。

要加的硬门禁(缺一不可):
  1. 禁止 partial/best-effort 进发布:preview_partial/submit_partial/pod 未就绪/冒烟有遗留 一律判 FAIL,不得进入 deploy;/ai-deploy-test force 对生产禁用。
  2. 机器人服务:发布前必须过「行为评测门禁」:每个 robot 仓建离线评测集(输入→期望行为)+自动打分,评测不通过禁止合入/发布 —— 替代「预览起来了就算过」。
  3. 「测不了就不许改」:CI 检测到无 pytest/vitest 可运行环境(如 PEP668/无网/Go 版本不符)时,禁止 Agent 改启动路径/核心逻辑代码(用户在 #621 已提)—— 不让"没跑过测试"的改动蒙混过关。
  4. 运行时验证纳入硬门禁:预览必须真实拉起 + 冒烟真通过(看 body/接口真返回,不看 pod 状态)才算过;发布前 required review + 真集成测试绿。
  5. 发布=合入边界:未合并的实现 PR 不得发布(现已有此拦截,保持);禁止绕过 PR 直接 commit 到生产分支。

这条不省小时,但避免生产事故 —— 优先级高于下面所有提效项。

做得好的凭什么(可复制经验)

  1. issue 正文自带规格/诊断 →「照抄落地」(最强因子):给根因、修复 SQL、文件:行号、参考 PR、参数表,AI 1 轮就对(#1128/#1226/#1013/#837)。
  2. 单点小改 + 量化验收:单文件/单接口、验收一句话或明确数字(500ms/两位小数/65%),改动面小难翻车(#1358/#1241/#1056)。
  3. 有现成同构模板可抄:参考已有 PR / playground 已存在 / 照清单逐项(#608 参考PR#117、#680、#1013)。
  4. 仓库 CLAUDE.md/skill 覆盖到位:AI 一次照约定做对(#1068 读了 datastat CLAUDE.md+skill、#1048 配置驱动原则)。

核心洞察:做得好=「人把 spec 写清 或 AI 已有领域知识」。所以两条腿:①要求 issue 写清可落地 spec ②给 AI 领域知识(I1)让它不必每次靠人喂。

想清楚后的一句话:模式 1/2/5/7/8(QA空转、跨轮记忆、越界、PR污染、引擎空跑)都是编排层的确定性缺陷,不是 AI 能力问题。「AI 缺领域知识」被高估了——一大半其实是「流水线每轮把上一轮学到的扔了,让人肉当记忆和边界检查器」。先补「记忆 + 边界 + 故障即停」这几个确定性编排缺陷,比投入任何「让 AI 更强」的事都快、都值。

Why 1:下一轮的 agent 根本没看到前几轮的纠正。

Why 2:每轮 /ai-develop-preview 都 spawn 一个全新 agent,喂给它的上下文只有 issue 正文 + 当前分支代码,没把「历次人工评论 + 历次纠正」累积喂进去。
↓ 为什么不累积喂?
Why 3:编排设计时把每轮当独立任务,假设「代码分支就是记忆」——但人工的纠正意图、为什么这么改、什么不能碰只在评论里,不在代码里。
🎯 根因:编排层没有「纠正记忆」——人肉在评论区当 agent 的记忆,而这个记忆没被喂回 agent。
🔧 怎么改:每轮 dev agent 的上下文注入「本 issue 全部人工评论 + 历次纠正摘要 + 已知禁止项清单(如『不要新建 X 文件』)」。纯编排改动,不依赖 AI 变聪明。
⚔️ 改了能解决吗?(对抗自问)
· 反问:评论太多会超上下文? → 用「纠正摘要 + 禁止项清单」而非全量塞。
· 反问:agent 看到了就一定不犯? → 大概率不犯(它本来就是没看到才犯的);残留是「看到了仍理解错」的少数。
结论:这是投入产出比最高、最具体、最不依赖『AI 变强』的一条——光这一条,可能砍掉 #840/#778/#927/#489/#921 这一批 issue 的过半轮次。

🔬 元根因:把三条串起来,再问最后一个为什么
↓ 为什么这三类问题会同时存在?
共同点:都把 AI 流水线当成「给个 issue 就自动出活」的零成本黑盒,没意识到它的产能高度依赖三个前置——①领域知识(AI 懂不懂)②spec 质量(人写清没)③平台稳定性(基建稳不稳)。
↓ 为什么会当成零成本黑盒?
最根本一条:组织把「AI 能做多少」的期望,建立在「零前置投入」的假设上——既不投入维护领域知识、也不规范 spec、还在平台不稳时就压业务。三个 TOP 问题都是这个假设的具体表现。
🎯 元根因:「AI 流水线 = 零成本黑盒」的错误预期——真实的 AI 产能 = f(领域知识, spec 质量, 平台稳定性),三者都要持续投入。
⚔️ 终极对抗:改了到底能不能解决?:能解决 80%+,但有前提——必须承认「AI 流水线不是零成本黑盒」,持续投入三件前置(知识/spec/平台)。反问:不投入行不行?→ 不行,再多 prompt 调优也是治标(#650/#927 靠调 prompt 救不回来,因为缺的是知识和稳定性,不是话术)。
一句话落地:先做投入产出比最高的 I1(领域知识,省 113h) 起步,再补平台稳定性(I3)与需求 spec(I5),三管齐下才是根治。
