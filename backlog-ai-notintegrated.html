🎯 最该干的:78 个仓有真实 PR 合入(在活跃开发)却没接入 AI Workflow(28 属已注册服务、50 个未归属)——榜首 datastat-server(合入 250)、whitebox(230),详见上方重点表。🔴 而已接入的 69 个仓里,38 个(55%)连 CLAUDE.md 都没有——AI 是「盲飞」着改的。往下问:

  • Why1 为什么接入了却没 CLAUDE.md? → CLAUDE.md 是人工逐仓写的,onboarding 时只在 umbrella 和少数核心仓(om-dataarts / datastat)写了,大量 dev 子仓没补。
  • Why2 为什么没补就有影响? → 这正是「AI 问题页」TOP1「缺领域知识」的物理证据:AI 真正 checkout 去改的是子仓,而 56% 的子仓里零仓级指引(表名/字段/约定/禁止项),只能每轮靠人喂 → 就是 #927/#650/#840 那种「同一纠正讲六七遍」。
  • Why3 为什么 skills 更稀缺(仅 11%)? → skills 是比 CLAUDE.md 更进阶的沉淀,只有被反复重度开发的少数核心仓才攒出来了;绝大多数仓连 CLAUDE.md 都没有,更谈不上 skills。
  • 根因 → 领域知识没落到「AI 真正会改的那个仓」里。服务级(umbrella)有文档 ≠ 子仓有文档;AI 的上下文喂的是子仓,断层就在这。
  • 怎么改 · 改了能不能解决(对抗) → 给这 38 个仓补最小 CLAUDE.md(表名+约定+禁止项)或让 umbrella 的 CLAUDE.md/skill 自动注入子仓 agent 上下文。能解决大头——因为它直接消灭「盲飞」;残留是「补了但写得不准」,需靠真跑校正。这条和问题页干预 I1(数据领域知识 skill + 各子仓 CLAUDE.md)是同一件事,这里给出了该补哪 38 个仓的确切清单(下表筛「接入AI的仓 + CLAUDE.md=—」即得)。

另:31 个「有 CLAUDE.md 却从没 AI PR」多是 umbrella / 文档 / 基建仓——AI 改的是它们的子仓,不直接对 umbrella 开 PR,属正常;个别工具库仓(被依赖的 SDK)可不强求 CLAUDE.md。

🎯 重点:有 PR 合入(真实在开发)却没接入 AI Workflow 的仓 —— 最该接入的目标
