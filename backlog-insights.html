⑥ 分析总结（文字）

场景分层 · 轮次多少的原因 · 待改动 · Action 时长

一、一共 8 类场景（+「其他/未分」兜底）

按任务性质把全部 issue 归成 8 个场景：前端看板/UI、数据/指标、机器人/自动化、CI/部署、安全/CVE、接口/API、会议、文档/流程，另有「其他/未分」作兜底。核心指标是「实现轮次」= 让 AI 把功能做对要反复戳 /ai-develop-preview 几轮（来自机器人流程标记 🟢[2]，含 accepted 自动触发），配合「关闭率」判断到底交付没交付。

二、轮次少 / 适中 / 多

🟩 轮次少（实现轮 ≤ 3.5）：会议(2.3)、接口/API(3.2)、安全/CVE(3.6)、CI/部署(2.0)、文档/流程(0)。
🟨 适中（3.5–7）：其他/未分(3.7)、数据/指标(6.8)。
🟥 轮次多（> 7）：前端看板/UI(10.3)、机器人/自动化(11.0)。

三、为什么少 / 为什么多（关键：别把「轮次少」都当好事）

「少」其实分两种，靠关闭率区分：

  • 真·省力（少 + 能交付）：会议(2.3 轮 / 关闭 50%)、接口/API(3.2 / 38%)。任务边界清晰、有明确验收（接口有契约、会议是定式），AI 一两轮就收敛。
  • 假·省力（少 + 交付不了）：CI/部署(2.0 轮 / 关闭 11%)、安全/CVE(3.6 / 0%)、文档/流程(0 / 0%)。轮次少不是因为简单，是 AI 根本接不动——试一两下没进展就停、最后靠人工兜底，所以既轮次低又几乎没关闭。这类是真短板。

「多」的原因是「复杂 + 缺自动反馈闭环」：

  • 前端看板/UI(10.3 轮，但关闭 53%)：AI 看不到渲染效果，配色/布局/交互对不对只能人肉看预览截图再喊下一轮 → 摩擦最大；好在最终能交付。
  • 机器人/自动化(11.0 轮，关闭仅 22%)：RAG/对话「回得好不好」没有确定性断言，得人反复主观评判 → 轮次爆炸且常常收不了尾。
  • 数据/指标(6.8，适中)：有一定可验性（产出结果集能对拍）但逻辑复杂，居中。

四、还需要做哪些改动

  • 轮次多的（UI / 机器人）→ 给 AI 装「自动反馈」：UI 场景做预览自动截图 + 视觉 diff 回灌，让 AI 自判页面对不对，把 10 轮压到 2–3 轮；机器人场景建离线评测集 + 自动打分，替代人肉评判。
  • 适中的（数据/指标）→ 加「产出对拍断言」：预期结果集比对，让 AI 有确定性反馈自收敛。
  • 假省力的（CI/部署、安全CVE、文档）→ 补「专用能力」，别硬塞通用 dev agent：CI 需 deploy agent + 环境自愈；CVE 需依赖图谱 + 补丁映射 + 升级回归；这些是能力缺口，不是调提示词能解决的。
  • 治理前提：约 390 个 issue 没打 target/project 服务标签，AI 渗透仅 18% —— 先补服务归属，才能按服务运营。

五、Action（流水线）时长分析

各阶段 GitHub Actions 单次运行时长（近 100 次成功 run 统计）：

  • 需求分析：中位 10 min（p90 15、max 21）
  • 开发实现：中位 20 min（p90 57、max 81）—— 绝对大头
  • 发布上线：中位 4 min（p90 18）
  • 事件分发：秒级（中位 0.3 min）

把轮次 × 单次时长换算成每个 issue 的「实现流水线机时」：前端看板 ≈ 10.3 轮 × 20 min ≈ 3.4 小时/issue；机器人 ≈ 11 轮 × 20 min ≈ 3.7 小时/issue；数据/指标 ≈ 6.8 × 20 ≈ 2.3 小时。这还没算人盯着看预览、写反馈的等待时间。所以降轮次不只是体验问题——直接省 Actions 机时 + 人力：前端看板若从 10 轮压到 3 轮，每个 issue 省约 2.3 小时流水线时间。优化优先级应对齐「轮次 × 时长 × issue 数」最大的场景，即前端看板 UI（66 个 issue × 3.4h）和机器人（36 × 3.7h）。
